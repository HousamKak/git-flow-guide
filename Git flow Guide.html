Introduction to Git

What is Version Control?

Version control is a system that records changes to files over time so that you can recall specific versions later. It allows you to revert files to a previous state, revert an entire project to a previous state, compare changes over time, see who last modified something, and more.

Key Git Concepts

Repository

A storage location for your project, containing all files and the history of changes made to those files.

Commit

A snapshot of your repository at a specific point in time, with a unique ID and message describing changes.

Branch

A parallel version of a repository that diverges from the main working project without affecting the main codebase.

Merge

The process of integrating changes from one branch into another, combining code from different development paths.

Pull Request

A way to propose changes from your branch to be reviewed by teammates before merging into another branch.

Clone

Creating a copy of a remote repository on your local machine to work with the entire project history.

Push & Pull

Push: Upload local changes to a remote repository.
Pull: Fetch and integrate changes from a remote repository.

Why Git is Helpful in Development Teams

1

Parallel Development

Multiple developers can work on the same project simultaneously without overwriting each other's changes.

2

Organized Feature Development

Features can be developed in isolation and merged when ready, allowing for structured development.

3

Full History & Accountability

Every change is tracked with the author's information, creating a clear history of who changed what and when.

4

Safe Experimentation

Developers can create branches to test new ideas without risking the main codebase, encouraging innovation.

Did You Know?

Git was created by Linus Torvalds in 2005 for development of the Linux kernel. The name "git" is British English slang for an unpleasant or stubborn person, which Linus jokingly applied to himself.

Git Flow Basics

What is Git Flow?

Git Flow is a branching model for Git that provides a robust framework for managing larger projects. It defines a strict branching structure designed around project releases, making parallel development more manageable.

Main
Production-ready code. Every commit on main is a new production release.
Dev
Main development branch containing latest delivered changes for the next release.
Feature
Branches off from develop and merges back to develop. Used for new features.
Release
Branches from develop and merges to main and develop. Prepares for a release.
Hotfix
Branches from main and merges to main and develop. For critical bugs in production.

Git Flow Commands

Here are the common Git Flow commands for different operations:

Operation Command
Initialize Git Flow git flow init
Start a feature git flow feature start FEATURE_NAME
Finish a feature git flow feature finish FEATURE_NAME
Start a release git flow release start RELEASE_VERSION
Operation Command
Finish a release git flow release finish RELEASE_VERSION
Start a hotfix git flow hotfix start HOTFIX_VERSION
Finish a hotfix git flow hotfix finish HOTFIX_VERSION

Manual Git Flow

You don't need the git-flow extension to use Git Flow. You can use standard Git commands to implement the workflow manually:

Create feature branch:

git checkout -b feature/feature_name develop

Merge feature into develop:

git checkout develop
git merge --no-ff feature/feature_name

Benefits of Git Flow

Structured Development

Clear roles for different branches help organize the development process.

Parallel Development

Multiple features can be developed simultaneously by different team members.

Release Management

Dedicated branches for releases make it easy to prepare and manage releases.

Quick Bug Fixes

Hotfix branches allow urgent fixes without disrupting ongoing development work.

Using JIRA with Git Flow

Integrating JIRA with Git Flow

JIRA is a powerful issue tracking and project management tool that works well with Git Flow. By linking your Git branches and commits to JIRA issues, you can create a traceable connection between your code changes and the tasks or stories they implement.

JIRA

  • Issue: PROJ-123
  • Summary: Implement login form
  • Status: In Progress

Git

  • Branch: feature/PROJ-123-implement-login
  • Commit: PROJ-123 Add login form validation

Naming Conventions

Feature Branches

feature/PROJ-123-feature-description

Includes the JIRA issue ID and a brief description.

Release Branches

release/1.2.0 or release/PROJ-sprint-5

Can be version-based or sprint/milestone-based.

Hotfix Branches

hotfix/1.2.1 or hotfix/PROJ-456-critical-bug

Includes version or JIRA issue ID of the critical bug.

Commit Messages

JIRA Issue References in Commits

Prefix Format

PROJ-123 Add login form

JIRA issue ID at the beginning of the commit message.

Suffix Format

Fix validation errors in registration #PROJ-123

JIRA issue ID with a hash symbol at the end.

JIRA Issue Transitions

1

Create Feature Branch

JIRA Status: "To Do" → "In Progress"

git checkout -b feature/PROJ-123-feature-name develop
2

Create Pull Request

JIRA Status: "In Progress" → "In Review"

Reference the JIRA issue in the PR description.

3

Merge Feature to Develop

JIRA Status: "In Review" → "Done"

git flow feature finish PROJ-123-feature-name

JIRA Integrations

JIRA DVCS Connector

Links JIRA to GitHub, Bitbucket, or GitLab repositories for seamless integration.

JIRA Automation

Create rules to transition issues based on Git events like branch creation or PR merges.

Smart Commits

Perform JIRA actions directly from commit messages:

PROJ-123 #comment Fixed bug #time 2h #resolve

Best Practices & Tips

Commit & Branching Best Practices

Write Good Commit Messages

  • Use the imperative mood: "Add feature" not "Added feature"
  • Keep the first line under 50 characters
  • Add more detailed explanation after a blank line if needed
  • Reference issue IDs (e.g., "PROJ-123")

Commit Frequently

  • Make small, focused commits that do one thing
  • Commit often to make changes easier to understand and review
  • Avoid "mega-commits" that change hundreds of files

Keep Feature Branches Small

  • Limit the scope of feature branches to specific functionality
  • Shorter-lived branches are easier to merge and have fewer conflicts
  • Aim to complete a feature branch within a few days to a week

Update from Develop Regularly

  • Regularly merge or rebase from develop to your feature branch
  • This reduces merge conflicts when completing your feature
  • Command: git checkout feature/my-feature && git merge develop

Code Review & Team Workflow

Pull Request Guidelines

  • Include a clear description of the changes
  • Reference the JIRA issue
  • Keep pull requests reasonably sized for effective review
  • Add tests and ensure all tests pass
  • Use checklists for common review items

Review Process

  • Assign at least one reviewer familiar with the code area
  • Address all feedback or explain why certain suggestions weren't implemented
  • Don't merge your own pull requests without review

Documentation

  • Document your team's Git Flow process
  • Create templates for pull requests and issues
  • Keep a changelog or release notes document

Automation

  • Use CI/CD pipelines to automate testing on branches
  • Set up branch protection rules to enforce code review
  • Automate JIRA transitions based on Git events

When to Deviate from Git Flow

Git Flow works well for projects with scheduled releases, but it may be too complex for some situations:

  • For continuous delivery workflows, consider GitHub Flow or Trunk-Based Development
  • For smaller teams or projects, a simplified workflow with just main and feature branches might suffice
  • Adapt the model to fit your team's specific needs

Semantic Versioning

What is Semantic Versioning?

Semantic Versioning (SemVer) is a versioning scheme that provides meaning to version numbers. Under this scheme, version numbers and the way they change convey meaning about the underlying code and what has been modified from one version to the next.

MAJOR
2
MINOR
4
PATCH
1

MAJOR

Incremented when you make incompatible API changes. This signals that users will need to adapt their code.

MINOR

Incremented when you add functionality in a backward-compatible manner. New features that don't break existing ones.

PATCH

Incremented when you make backward-compatible bug fixes. Improvements that don't add features but fix problems.

SemVer Examples & Implementation

Common Version Changes

Change Example When to Use
Patch 1.2.3 → 1.2.4 Bug fixes
Minor 1.2.3 → 1.3.0 New features
Major 1.2.3 → 2.0.0 Breaking changes
Pre-release 1.2.3-alpha.1 Testing versions

Tag-Based Versioning in Git

Git tags are perfect for marking release versions:

git tag -a v1.2.3 -m "Release version 1.2.3"
git push origin v1.2.3

Automating Version Management

Several tools can help automate SemVer:

  • npm/yarn version: For JavaScript projects
  • bumpversion: A Python utility
  • GitVersion: Calculates versions from Git history

SemVer with Git Flow

1

Main Branch (v1.0.0)

Your stable release with a MAJOR.MINOR.PATCH version tag.

2

Hotfix Branch

Increments PATCH version (v1.0.0 → v1.0.1) for critical bug fixes in production.

3

Release Branch

Increments MINOR version (v1.0.1 → v1.1.0) for new features ready for release.

4

Major Version Updates

Increments MAJOR version (v1.1.0 → v2.0.0) for API-breaking changes.

Dependency Management

Version Ranges in Dependencies

Syntax Example Meaning
Exact "express": "4.17.1" Exactly this version
Caret (^) "express": "^4.17.1" 4.17.1 ≤ x < 5.0.0
Tilde (~) "express": "~4.17.1" 4.17.1 ≤ x < 4.18.0

When to Release Version 1.0.0

Version 1.0.0 defines the public API. It indicates your software is:

  • Ready for production use
  • Has a stable API that users can depend on
  • Future versions will follow SemVer guidelines

SemVer Rules

  • Once a version is released, its contents must not be modified
  • Pre-release versions (alpha, beta) indicate instability
  • Major version zero (0.y.z) is for initial development

SemVer in Real World Projects

Many popular projects follow SemVer with their own interpretations:

Linux Kernel

Uses a different scheme (major.minor.patch-build)

Angular

Strictly follows SemVer with time-based major releases

React

Generally follows SemVer, with some flexibility during pre-1.0
